# DC Drift

**An AI agent that detects design–code drift before release.**

DC Drift compares approved Figma designs against shipped code, classifies
every divergence with a deterministic ruleset, and produces a Drift Score
with role-specific reports — for PMs, engineers, and designers — in under
60 seconds. It replaced a 30–45 minute manual QA process in my own
release workflow.

Built with Claude Code, Figma MCP, n8n, and React.

📖 Full case study: [anupamatiwari.com](https://anupamatiwari.com) · Live verification report generated by the pipeline available there.

---

## The problem

Between design approval and release, things drift: a token gets overridden,
a component state goes missing, spacing compresses. Nobody catches it until
QA — or production. There is no fast, reliable way to verify that the
implemented UI still matches the approved design *intent*. Visual regression
tools compare yesterday's build to today's; they can't tell you whether
either matches what was approved.

## What it does

```
Approved Figma frame (source of truth)
        ↓
Figma MCP — actual tokens, spacing, structure (not screenshots)
        ↓
Claude Code agent — governed by .dc-drift/rules.md
        ↓
Dual-layer analysis
  ├─ Structural audit: token usage, type scale, component states (from code)
  └─ Visual regression: frame vs implementation screenshot (Claude Vision)
        ↓
Rule engine → severity (HIGH / MEDIUM / LOW) + confidence per finding
        ↓
Scoring: structural = checks passed / checks run · visual = 100 − (20×H) − (10×M) − (5×L) · combined = min
        ↓
Release gate: ≥80 GO · 60–79 REVIEW · <60 HOLD
        ↓
Three reports from one findings set: PM decision · Engineer checklist · Designer confidence
```

**Why deterministic scoring instead of conversational AI?** Early versions
let the LLM describe issues freely — the same issue was described
differently across runs, making release decisions impossible to
standardize. Every finding now maps to a rule, a severity, a penalty, and
a confidence score. Same input, same output, every time. In release
decisions, consistency *is* trust.

**Why confidence thresholds?** Vision-based comparison can misread
rendering artifacts. Findings below 0.75 confidence are flagged for human
review; below 0.50 they're suppressed. The system is honest about what it
doesn't know.

## Verification history — real runs, real receipts

**Run 1 (baseline): 0/100.** The first execution against a bare Next.js
scaffold correctly reported a *complete implementation gap* — 6 HIGH,
3 MEDIUM, 1 LOW findings — rather than hallucinating a comparison with a
screen that didn't exist. Detecting absence honestly is as important as
detecting drift. Full report: [`.dc-drift/reports/run-1-2026-03-13-baseline.md`](.dc-drift/reports/run-1-2026-03-13-baseline.md)

**Run 2 (post-implementation, dual layer): 72 structural / 95 visual — Δ23.**
After the screen was built, the visual layer scored 95/100 (one LOW
finding, a badge position, 0.91 confidence, ~15-min fix) — but the
structural audit passed only **72 of 100 checks**: missing CTA states,
absent payment validation, breakpoints broken at tablet widths, token
debt. Combined score = min(structural, visual) = **72, REVIEW REQUIRED**.
That 23-point divergence is the argument for dual-layer verification — a
screenshot tool would have said "looks good." Full report:
[`.dc-drift/reports/run-2-2026-03-14-dual-layer.md`](.dc-drift/reports/run-2-2026-03-14-dual-layer.md)

The pair demonstrates the point of the system: it measures progress, and
its dual-layer design exists because structural analysis and visual
analysis catch different defects — missing states, token debt, and broken
breakpoints pass a screenshot comparison and break in production.

**The 30-second architecture answer:** Layer 1 thinks (Claude Code +
Figma MCP + rules.md). Layer 2 coordinates (n8n triggers, routing,
outputs). Layer 3 communicates (dashboard and role-specific reports).

## Current state — honest status

The system works as two paths that aren't yet connected:

- **Path A (depth):** Claude Code + Figma MCP + full codebase access.
  324k characters of design context, real token values. Runs manually.
- **Path B (automation):** n8n pipeline — trigger → Figma API → dual-layer
  analysis → Drift Score → Google Docs report. Runs end-to-end without
  anyone touching it, but can't read the codebase.

**Next step (V3):** a webhook bridge letting n8n trigger Claude Code as a
subprocess — Path A's depth with Path B's automation. One pipeline, one
source of truth. No new AI model, no new scoring logic; the bridge just
connects what already works.

## Setup

1. Connect Figma MCP to Claude Code:
   ```bash
   claude mcp add --transport http figma-remote-mcp https://mcp.figma.com/mcp
   ```
   Verify with `/mcp`.
2. Keep credentials in environment variables — never in files:
   ```bash
   export FIGMA_TOKEN=<your token>   # Figma → Settings → Security
   ```
3. Run a verification from Claude Code:
   ```
   Verify [route] against the approved Figma frame "[frame name]".
   Use Figma MCP for design context. Apply .dc-drift/rules.md.
   Return findings, severities, confidence, Drift Score, and all 3 reports.
   ```

## What building this taught me

- **Better data beats better prompts.** Early versions fed the agent frame
  names and hallucinated confidently. Feeding actual hex values, font
  sizes, and spacing from the Figma API ended the hallucinations.
- **Transparency earns trust.** Every finding has a confidence score; every
  score has a formula; every gate has a threshold. Black-box AI
  recommendations get ignored — transparent ones get acted on.
- **Design the output for its consumer.** Same 13 findings, three views:
  a PM gets a decision, an engineer gets a checklist, a designer gets
  confidence scores. That's not formatting — that's a product decision.
